# Rutsubo — Build Prompt

## プロジェクト概要

**Rutsubo（るつぼ）** は、AIエージェントが競い合い、自然淘汰される経済圏を作るプロトコルです。
「エージェントが試される場（crucible）」というコンセプトで、強化学習の探索・活用のロジックを使い、
パフォーマンスの高いエージェントが自動的に採用され、Solana上で報酬を受け取る仕組みです。

---

## 背景・動機

### 解決する問題
- 複数のAIエージェントサービスにタスクを投げたいが、「どのエージェントが本当に貢献したか」がわからない
- 貢献度の評価が主観的・不透明で、公平な報酬分配ができない
- タスクの失敗原因が「プロンプトが悪かったのか」「分解が雑だったのか」「実行が悪かったのか」わからない

### 既存OSSとの差別化
- **LangGraph / CrewAI** などのオーケストレーションフレームワーク：タスク分解・実行はできるが、貢献度評価→報酬分配→自律的な採用ロジックが存在しない
- **agentregistry（Solo.io）**：エージェントの登録・管理はできるが、パフォーマンスベースの自動採用・淘汰がない
- **オンチェーンreputationの研究**（Medium等）：概念は存在するが、リアルタイムの報酬分配・因果連鎖の可視化に繋がっていない

**Rutsuboの新規性：**
評価ループ・報酬分配・オンチェーン永続化・自律的な採用ロジックを一気通貫で繋いだものが存在しない。

## アーキテクチャ

### 全体フロー

```
タスク定義（ユーザーのプロンプト）
    ↓
コーディネーターエージェント（サブタスクに分解・割り当て）
    ↓
ワーカーエージェントA / B / C（サブタスク実行）
    ↓
レビュアーエージェント（非対称損失関数で品質評価）
    ↓
メタレビュアー（全レイヤーの評価精度を統合判定）
    ↓
Solana オンチェーン決済（貢献度スコアで自動分配）
    ↓
[フィードバックループ] 履歴が蓄積 → 次回の採用確率・信頼スコアに反映
```

### 評価レイヤー（重要）

各レイヤーを独立して評価する。評価結果はオンチェーンに記録される。

| レイヤー | 評価内容 |
|---|---|
| タスク定義 | プロンプト・要件の曖昧さ、明確さ |
| コーディネーター | サブタスクの粒度・割り当ての適切さ |
| ワーカー | サブタスクの実行品質・速度 |
| レビュアー | 評価の精度（見逃し・過検出のバランス） |
| メタレビュアー | レビュアー自身の過去評価精度 |

**因果連鎖の可視化が重要：**
「タスク定義が曖昧 → 分解が雑になる → ワーカーの成果が低い」という責任の所在を
どのレイヤーにあるかオンチェーン記録から逆引きできること。

### 貢献度評価の設計思想

LayerX社のブログ（AIエージェントのHuman-in-the-Loop評価 | https://tech.layerx.co.jp/entry/2026/04/01/150000 ）を参考に、
**非対称損失関数**を採用する：

- **見逃し（下振れ）は非線形に重くペナルティ** → 重要なタスクの失敗は報酬が急激に減る
- **過検出（上振れ）は線形コスト** → 不要な介入は軽微なペナルティ
- タスクのリスクレベルに応じてパラメータを動的に変える

### 採用ロジック：LLMベースの意味的マッチング + ε探索

UCBのような統計的手法だけでは「タスクの内容」と「エージェントの得意領域」のミスマッチを防げない。
例えば「コード生成タスク」と「文章要約タスク」では最適なエージェントが全く異なるため、
**コーディネーターLLMがタスクの意味的内容を理解した上でエージェントを選ぶ**設計にする。

#### コーディネーターLLMへの入力
```
- タスクの内容・難易度・ドメイン（例：コード生成、データ分析、文章要約）
- 各エージェントのプロフィール（登録時の説明・得意領域）
- 各エージェントの過去実績サマリー（オンチェーンから取得した信頼スコア・タスク種別ごとの成績）
- 現在の探索率パラメータ ε
```

#### コーディネーターLLMの判断
LLMが「このタスクにはAとCが良さそう、ただBはまだ試したことがないから入れてみる」と
自律的に判断して選択する。純粋な統計ではなく、**タスクとエージェントの意味的な相性**を考慮できる。

#### 探索の保証
LLMに完全に任せると収束しすぎる（実績のあるエージェントしか使わなくなる）リスクがある。
そのため、**εの確率でランダムに実績の少ないエージェントを強制的に選ぶ**保険を残す。
εはシステム全体のトランザクション数が増えるにつれて小さくしていく（焼きなまし的に）。

採点基準への対応：
- **Innovation 40%**：評価ループ・報酬分配・オンチェーン永続化・自律採用を一気通貫で繋いだものが存在しない
- **Agentic Sophistication 30%**：AIが採用・解雇・評価・分配を自律的に判断する
- **Traction 30%**：ハッカソン参加者のエージェントを実際に登録・トランザクション発生

```python
  "wallet_address": "Solanaウォレットアドレス",
  "endpoint": "エージェントのAPIエンドポイント"

```

### Step 2：タスク投入とコーディネーター

```python
POST /tasks
{
  "prompt": "ユーザーのタスク内容",
  "budget": 0.1  # SOL or USDC
}
```

コーディネーターが：
1. タスクを受け取る
2. 難易度・リスクレベルを自律判定（LLMで推定）
3. 登録済みエージェントの中から候補を選ぶ（ε-greedy）
4. サブタスクに分解して割り当てる

### Step 3：評価と分配

各ワーカーの成果をレビュアーエージェントが評価し、
スコアに応じてx402-solanaでウォレットに自動送金。

## ワーカーエージェントの設計方針

### 内部実装はブラックボックスでよい

Rutsuboがワーカーエージェントに要求するのは以下のみ：
- タスクを受け取るAPIエンドポイント（`POST /execute`）
- 結果テキストを返すこと
- Solanaウォレットを持っていること

内部でRAGを使おうが、ファインチューニングモデルを使おうが自由。
むしろ「RAGを使った専門エージェントが特定ドメインで強い」という**エージェントの専門化・差別化**が
自然に生まれることがRutsuboの面白さ。

### RAGを使ったワーカーエージェントの例（サンプル実装として用意する）

```python
# rag_agent.py — RAG付きワーカーエージェントのサンプル
from anthropic import Anthropic
import chromadb  # or any vector DB

class RAGWorkerAgent:
    def __init__(self, knowledge_base_path: str):
        self.client = Anthropic()
        self.db = chromadb.Client()
        self.collection = self.db.get_or_create_collection("knowledge")
        # ドキュメントをロードしてインデックス化
        self.load_knowledge(knowledge_base_path)

    def execute(self, subtask: str) -> str:
        # 1. 関連ドキュメントを検索
        results = self.collection.query(query_texts=[subtask], n_results=3)
        context = "\n".join(results["documents"][0])

        # 2. コンテキストを使ってLLMで回答生成
        response = self.client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1000,
            messages=[{
                "role": "user",
                "content": f"Context:\n{context}\n\nTask:\n{subtask}"
            }]
        )
        return response.content[0].text
```

## テスト戦略

### ラージテスト（Traction証明）の方法

実際のユーザーがいない状態でトランザクション数を増やすための方針：

#### 方法1：シミュレーション環境（自動テスト）
品質が異なる複数のダミーエージェントを用意して大量タスクを自動投入する。
「良いエージェントの採用率が時間とともに上がっていく学習曲線」をグラフで示す。

```python
# simulation.py — ラージテスト用シミュレーター
import asyncio
import random

# 品質レベルが異なるダミーエージェントを定義
DUMMY_AGENTS = [
    {"name": "HighQualityAgent", "quality": 0.9},   # 優秀
    {"name": "MediumAgent",      "quality": 0.6},   # 普通
    {"name": "PoorAgent",        "quality": 0.3},   # 低品質
    {"name": "NewAgent",         "quality": None},  # 未知（探索対象）
]

async def run_simulation(n_tasks: int = 100):
    for i in range(n_tasks):
        task = generate_random_task()
        await submit_task(task)
        await asyncio.sleep(0.5)
```

審査員への見せ方：
- x軸：タスク数（時間経過）
- y軸：各エージェントの採用確率
- HighQualityAgentの採用率が自然に収束していく様子を可視化

#### 方法2：他のハッカソン参加者のエージェントを繋ぐ（本物のTraction）
Discordで早めに声をかけて、参加者の実エージェントを登録してもらう。
「登録すればdevnetのSOLで報酬が得られる」という動機を提供。
→ これが最もリアルなトランザクションになる。

#### 方法3：自分で複数エージェントを実装して競わせる
RAGなし版 vs RAGあり版 vs ファインチューニング版など、
意図的に違う実装のエージェントを複数用意してRutsuboに登録し競わせる。
「ドメイン特化エージェントが特定タスクで強い」という動作を実証する。

- Solana/cryptoの経験はほぼゼロなので、**x402-solanaやSolana Agent Kitのサンプルコードを積極的に活用**すること
- スマートコントラクト（Rust/Anchor）は書かない。SDKで完結させる
- まず動くものを優先。きれいなコードより動くコードを先に
- テスト用のdevnetを使うこと（本物のSOLは使わない）
